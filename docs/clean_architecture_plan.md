# Clean Architecture Migration Plan

This document tracks the staged migration of the wilayah-indonesia project toward the Clean Architecture blueprint described in `GUIDE.md`. Each phase captures intent, concrete tasks, and ownership notes so the work can be delivered incrementally without breaking existing behaviour.

## Phase 1 – Baseline & Target Map

### Current Architecture Inventory

| Area | Key Components | Responsibility Snapshot | Notes |
| --- | --- | --- | --- |
| Entrypoints | `cmd/api/main.go` | Boots Fiber HTTP server, wires DuckDB connection, registers routes directly with services | No configuration abstraction; logging, env handling inline |
| Entrypoints | `cmd/ingestor/main.go` | Executes data ingestion SQL scripts, builds DuckDB tables, manages FTS indexes | Performs file IO, transformation, and schema management in one binary |
| HTTP Layer | `internal/api/handler.go` | Fiber handlers parsing queries, calling `pkg/service.Service` | No separation between transport and application layers |
| Business Logic | `pkg/service/service.go`, `pkg/service/errors.go` | Region search logic (FTS queries, option normalisation), domain models, error typing | Serves both HTTP and potential future adapters |
| Data Access | DuckDB SQL executed inside `pkg/service` | Direct SQL assembly and execution, schema introspection, FTS detection | Lacks explicit repository abstraction; queries embedded in service |
| Data Refresh | `cmd/ingestor` helper functions (`execSQLFromFile`, `removeMySQLSyntax`) | Prepares dataset and FTS indexes | Could become a gateway/use case pair |
| Tooling & Data | `data/`, `scripts/`, `regions-api/` | SQL sources, fetch scripts, API samples | Shared across runtimes |

### Domain Glossary

- **Region** — Core entity representing Indonesian administrative areas enriched with postal codes and BPS metadata.
- **RegionBPS** — Optional per-level BPS metadata (code + name) linked to a region.
- **SearchOptions** — Normalised settings governing result limits, inclusion of BPS data, and score presentation.
- **SearchScores** — Scoring artefacts generated by DuckDB FTS queries, optional in API responses.
- **BPS Mapping** — Dataset aligning Kemendagri and BPS hierarchies, created during ingestion and reused by search.
- **FTS Indexes** — DuckDB full-text search indexes over Dagri (`regions`) and BPS (`regions_bps`) corpora to power fuzzy matching.

### Target Clean Architecture Layout

| Clean Architecture Layer | Planned Package(s) | Migration Source | Highlights |
| --- | --- | --- | --- |
| Delivery (HTTP) | `internal/delivery/http` (controllers, routes, middleware) | `internal/api/handler.go`, portions of `cmd/api` | Thin handlers converting HTTP ↔ models, delegating to use cases |
| Delivery (Jobs) | `internal/delivery/worker` (ingestor orchestration) | `cmd/ingestor` | Allows reuse of ingestion logic via structured commands |
| Configuration / Wiring | `internal/config` | `cmd/api`, `cmd/ingestor` | Central bootstrap for logger, Fiber, DuckDB, feature toggles |
| Use Cases | `internal/usecase/region`, `internal/usecase/ingestion` | `pkg/service` (search) & `cmd/ingestor` orchestration | Encapsulate application rules, validation, transactions |
| Repository | `internal/repository/region` (DuckDB impl under `internal/repository/duckdb`) | SQL inside `pkg/service` | Provide interfaces + adapters for DuckDB queries |
| Entities | `internal/entity` | Structs from `pkg/service.Region`, SQL schema knowledge | Persistence representations aligned with DuckDB tables |
| Models / DTO | `internal/model` | Request parsing currently in handlers, response structs in `pkg/service` | Transport layer contracts for HTTP and jobs |
| Gateways | `internal/gateway/bps`, future caches | Not yet implemented | Placeholder for outbound integrations (BPS API, caching) |
| Validation & Errors | Shared packages under `internal/shared` or reuse existing | `pkg/service/errors.go`, inline checks | Consistent error surface between delivery layers |
| Tests | `test/` (integration), package-local `*_test.go` | `pkg/service/service_test.go`, scripts | Reorganise to mirror new packages with unit + integration coverage |

Phase 1 deliverables complete the baseline understanding required to start refactoring.

## Phase 2 – Foundation Packages

**Goal:** Establish the package skeleton that mirrors Clean Architecture boundaries without migrating logic yet.

**Tasks**
- Create `internal/config` with initial constructors:
  - `config.Options` struct capturing env vars (`DB_PATH`, `PORT`, feature flags).
  - `NewLogger()`, `NewDuckDB(options)`, `NewFiber(options)`, each returning interfaces for defer-friendly teardown.
  - `BootstrapHTTP(opts)` returning assembled Fiber app, repositories, and use cases (initially placeholders).
- Introduce domain persistence structs under `internal/entity`:
  - `entity.Region`, `entity.RegionBPS`, `entity.RegionFTS` mirroring DuckDB columns.
  - Ensure tags remain aligned with database column names.
- Define transport models in `internal/model`:
  - Request query options (`model.SearchRequest`), response types (`model.RegionResponse`, `model.SearchResponse`), error envelopes.
  - Shared pagination/search option helper functions.
- Draft repository contracts in `internal/repository`:
  - `RegionRepository` interface with `Search`, `SearchByDistrict`, etc.
  - `internal/repository/duckdb` implementation stubs returning `ErrNotImplemented` to be filled during Phase 6.
- Provide shared error definitions under `internal/shared/errors` (wrap existing service errors for reuse).
- Update build tooling (`Makefile`) only to include new packages in `go fmt`/`go test` targets once code lands.

**Artifacts**
- Empty or placeholder structs/interfaces compiled to prevent regressions.
- Documentation comments referencing migration steps for future contributors.

## Phase 3 – Use Case Layer

**Goal:** Carve business rules out of `pkg/service` into dedicated use case packages while maintaining behaviour through interfaces.

**Tasks**
- Stand up `internal/usecase/region` package:
  - Define `type RegionUseCase interface` exposing the six search entry points currently on `pkg/service.Service`.
  - Implement `regionUseCase` struct that accepts `repository.RegionRepository` and pure helpers for option validation and mapping.
  - Port `normalizeOptions`, query validation, and error creation from `pkg/service` using `internal/shared/errors`.
- Extract cross-cutting helpers under `internal/usecase/shared`:
  - `OptionNormalizer` component encapsulating limit bounds and defaults for reuse by other use cases.
  - `ResultMapper` functions converting `entity.RegionWithScore` to `model.RegionResponse` while honouring BPS/score toggles.
- Replace embedded SQL by invoking repository interfaces and handling feature-toggle logic (include BPS, scores) within the use case.
- Ensure all service-level structs (`SearchQuery`, `SearchOptions`, `Region`, `RegionBPS`, `SearchScores`) now live in `internal/model` and are consumed by controllers/use cases.
- Provide constructor `NewRegionUseCase(repo RegionRepository, opts RegionUseCaseOptions)` that surfaces tuning knobs (default limit, BPS toggle, logger) and returns the interface implementation.
- Retire `pkg/service` entirely once the new use case compiles; add a short-lived façade only if required for build order but remove before exiting the phase.
- Document context usage, error propagation, and logging expectations within the use case package README or doc comment.

**Artifacts**
- Fully implemented `internal/usecase/region` with table-driven unit tests covering input validation, option normalisation, and repository error translation.
- Updated `internal/model` definitions referenced by the use case and future controllers.
- Confirmation that `pkg/service` is removed or reduced to a deprecation stub scheduled for deletion.

## Phase 4 – HTTP Delivery Refactor

**Goal:** Rebuild the HTTP layer as Clean Architecture delivery adapters that consume the new use case interfaces.

**Tasks**
- Scaffold delivery layer packages:
  - `internal/delivery/http/router` handling route registration and dependency injection (`RegisterHTTP` returning configured `*fiber.App`).
  - `internal/delivery/http/region` controllers that translate query params to `model.SearchRequest` and call the use case.
  - `internal/delivery/http/middleware` for logging, recovery, and feature toggles; leverage Fiber’s middleware ecosystem carefully.
- Build reusable response helpers under `internal/delivery/http/response` to unify JSON success/error payloads and HTTP status codes.
- Update `cmd/api/main.go` to defer all wiring to `internal/config.BootstrapHTTP`, keeping only config loading, signal handling, and `app.Listen`.
- Maintain endpoint parity:
  - `/v1/search`, `/v1/search/district`, `/v1/search/subdistrict`, `/v1/search/city`, `/v1/search/province`, `/v1/search/postal/:postalCode`, and `/healthz` all routed via new controllers.
  - Provide request/response mapping tests to guarantee compatibility with existing clients (`regions-api/`).
- Integrate validation (either Fiber query validation or go-playground/validator) at the controller boundary; document default behaviours in package comments.
- Ensure error translation uses `internal/shared/errors` to provide consistent JSON responses and HTTP statuses.

**Artifacts**
- New delivery packages with functioning controllers and middleware, backed by unit tests or handler-level integration tests.
- Simplified `cmd/api/main.go` delegating to the bootstrapper.
- Compatibility checklist demonstrating parity with previous handlers.

**Artifacts**
- New delivery packages with gofmt’d skeletons.
- Updated entrypoint using bootstrapper.
- Markdown note summarising route parity checks.

## Phase 5 – Ingestor Alignment

**Goal:** Treat the data refresh workflow as another delivery adapter that leverages shared configuration, repositories, and use cases.

**Tasks**
- Build `internal/usecase/ingestion` package:
  - Encapsulate SQL script execution, transformation queries, FTS setup, and clean-up behaviour (`Refresh(ctx, opts)` entry point).
  - Accept collaborators via interfaces: `SQLLoader`, `SQLExecutor`, `FTSManager`, `DatasetChecker` to ease testing.
  - Port logic from `cmd/ingestor` (`execSQLFromFile`, `removeMySQLSyntax`, transformation query) into reusable gateways or repository helpers invoked here.
- Create delivery adapter under `internal/delivery/worker/ingestor`:
  - Provide command runner that maps CLI flags/env to `IngestionOptions` and logs each stage using structured logging.
  - Support dry-run mode and targeted refresh (e.g., skip BPS reload) if needed for future operations.
- Extend `internal/config`:
  - Introduce `BootstrapWorker` returning shared dependencies (logger, DuckDB connection, ingestion use case) for `cmd/ingestor`.
  - Centralise environment parsing (paths, concurrency knobs, feature flags).
- Rebuild `cmd/ingestor/main.go` so it only parses CLI args, invokes `BootstrapWorker`, and runs the worker adapter with graceful shutdown.
- Ensure ingestion layer emits consistent errors leveraging `internal/shared/errors` and logs metrics for start/finish/duration per major step.
- Add TODOs/placeholders for future BPS API integration under `internal/gateway` so the ingestion flow can pivot to live fetches later.

**Artifacts**
- `internal/usecase/ingestion` and worker delivery package with unit tests covering error scenarios (missing file, SQL failure).
- Updated `cmd/ingestor` that delegates to the new architecture.
- Notes on observability hooks and configuration defaults.

**Artifacts**
- Worker delivery package ready for CLI wiring.
- Use case with integration hooks (allowing invocation from CLI or future API endpoint).

## Phase 6 – Repository & Gateway Enhancements

**Goal:** Replace inline SQL with repository abstractions and prepare outbound gateways for external systems.

**Tasks**
- Finalise repository contracts:
  - Enrich `internal/repository/region` with context-aware methods returning domain entities plus metadata (scores, total count when available).
  - Introduce additional interfaces as needed (e.g., `AdminRepository` for ingestion DDL/DML operations).
- Implement DuckDB adapters under `internal/repository/duckdb`:
  - Translate current SQL from `pkg/service` and `cmd/ingestor` into prepared statements.
  - Handle optional joins for BPS data and scoring toggles efficiently.
  - Provide helper scanners that map result rows into `entity.RegionWithScore`.
- Add error translation helpers turning DuckDB-specific errors into `internal/shared/errors` codes; include context on query type and parameters for logging.
- Seed gateway layer with extensibility points:
  - `internal/gateway/bpsapi` placeholder returning `ErrNotImplemented` but documenting required interface for future external syncs.
  - `internal/gateway/cache` interface in case caching is introduced.
- Share ingestion SQL execution through repositories/gateways so the worker use case can reuse the same primitives (transformation query, FTS creation, cleanup).
- Document expectations via package-level comments: context propagation, connection management, testing strategy (DuckDB in-memory fixtures).

**Artifacts**
- DuckDB repository implementations passing unit/integration tests.
- Gateway interfaces with stubs checked in and referenced by use cases.
- Documentation on repository behaviour and testing approach.

**Artifacts**
- Concrete repository implementations tested against DuckDB (in-memory or fixture database).
- Gateway interfaces with placeholder implementations ready for future expansion.

## Phase 7 – Testing & Contracts

**Goal:** Ensure the new architecture is protected with a balanced mix of unit, integration, and contract tests.

**Tasks**
- Relocate and expand unit tests:
  - Port `pkg/service/service_test.go` into `internal/usecase/region` with repository fakes covering success and error paths.
  - Add boundary-case tests for option normalisation, BPS toggle behaviour, and error translation.
- Build integration layers:
  - `test/api` exercising full HTTP stack via Fiber test client and `config.BootstrapHTTP` with in-memory DuckDB.
  - `test/worker` ensuring ingestion flow processes SQL dumps end-to-end (guarded behind build tags if slow).
- Extend tooling:
  - Update `Makefile` to include new test commands (`test-api`, `test-worker`) and ensure `make test` covers package + integration suites.
  - Set up CI pipeline notes to run DuckDB in-memory to avoid filesystem permissions issues.
- Contract verification:
  - Validate responses against existing `regions-api/` samples or OpenAPI schema.
  - Consider adding Golden files for representative search queries.

**Artifacts**
- Comprehensive test suite mirroring architecture layers.
- Updated Makefile / CI instructions.
- Notes on test data management and environment requirements.

**Artifacts**
- Comprehensive test directories mirroring package structure.
- CI notes/checklist for verifying all targets before merging.
